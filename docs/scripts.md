# Scripts

## Purpose

The `scripts` folder contains helper PowerShell scripts for creating EF Core migrations:

- `add-basket-migration.ps1`
- `add-order-migration.ps1`
- `add-logistics-migration.ps1`

Each script is scoped to one service and generates the next sequential migration name automatically.

## What Each Script Does

For its service, each script:

1. Resolves the repository root from `PSScriptRoot`.
2. Locates the infrastructure project.
3. Locates the API startup project.
4. Ensures the `Migrations` folder exists.
5. Scans existing migrations and extracts the highest three-digit service version.
6. Generates the next migration name in this format:
   - `basket_001`
   - `order_001`
   - `logistics_001`
7. Runs `dotnet ef migrations add ... --output-dir Migrations`.

The timestamp prefix is still generated by EF Core, so the final file name becomes something like:

- `20260331120000_basket_003.cs`

## Preview Mode

All scripts support:

```powershell
.\scripts\add-basket-migration.ps1 -Preview
```

Preview mode prints the generated command and migration name without creating files.

## Environment Setup Inside the Scripts

Before invoking `dotnet ef`, the scripts set:

- `DOTNET_SKIP_FIRST_TIME_EXPERIENCE=1`
- `DOTNET_CLI_HOME=<repo>/.dotnet`

That keeps CLI state local to the repository.

## Usage

Examples:

```powershell
.\scripts\add-basket-migration.ps1
```

```powershell
.\scripts\add-order-migration.ps1 -Preview
```

```powershell
.\scripts\add-logistics-migration.ps1
```

## Important Behavior Note

The comments inside the scripts say migrations are applied manually at startup, but the current API projects call `Database.MigrateAsync()` during application startup for all three services.

That means the current runtime behavior is:

- these scripts create reviewed migration files
- the APIs apply pending migrations automatically when they start

The code should be treated as the source of truth unless that startup behavior changes.

## Limitations

- The scripts create migrations only; they do not generate SQL scripts.
- They assume the standard project layout under `src/Services/...`.
- They are designed for interactive local use and pause before exit.
